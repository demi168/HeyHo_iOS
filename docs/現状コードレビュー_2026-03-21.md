# 現状コードレビュー（2026-03-21）

対象: `heyho_ios` / `functions` / `firestore.rules`  
観点: セキュリティ、メンテナンス性  
方針: **現時点の実装のみ**を根拠に記載（過去状態は含めない）

---

## 優先度定義

- **P0**: 悪用や情報漏えいに直結しやすい
- **P1**: 仕様逸脱・運用障害・長期的な保守コスト増を招く
- **P2**: 早急ではないが、放置で劣化しやすい

---

## セキュリティ

### SEC-001（P0） `users` の `get` で機微情報を取得できる

- **問題**: `users/{userId}` は認証済みなら `get` 可能。`AppUser` に `fcmToken` と `inviteCode` が同居している。
- **影響**: UID が分かれば他ユーザーの通知トークンや招待コードを取得でき、悪用余地が残る。
- **根拠ファイル**:
  - `firestore.rules`
  - `heyho_ios/Models/User.swift`
  - `heyho_ios/Services/FirestoreService.swift`

### SEC-002（P0） 友だち関係を片側操作で成立させられる

- **問題**: `users/{userId}/friends/{friendId}` への `write` が `isOwner(userId) || isOwner(friendId)`。
- **影響**: 相手同意なしで友だち関係（相手側ドキュメントを含む）を作れてしまう。
- **根拠ファイル**:
  - `firestore.rules`
  - `heyho_ios/Services/FirestoreService.swift`（`addFriend`）

### SEC-003（P1） 招待コードが 6 桁数字で推測耐性が低い

- **問題**: コード空間が 100 万通りのみ。`inviteCodes/{code}` は `list` 拒否だが `get` は許可。
- **影響**: 試行回数次第で有効コードの発見確率が現実的になりうる。
- **根拠ファイル**:
  - `firestore.rules`
  - `heyho_ios/Services/FirestoreService.swift`（`generateInviteCode`, `getUserIdByInviteCode`）

### SEC-004（P2） Cloud Functions 側の入力バリデーションが薄い

- **問題**: `onHeyHoCreated` は `snap.data()` を型断定で利用し、欠損/型不正の明示ガードがない。
- **影響**: ルール変更や管理操作で不正データが混入した場合、通知文言の崩れや障害解析の難化につながる。
- **根拠ファイル**:
  - `functions/src/index.ts`
  - `functions/src/notificationTemplates.ts`

## メンテナンス性

### MNT-001（P1） 使えない検索機能が UI 導線に残っている

- **問題**: `searchUsers` に「list 拒否で動作しない」と注記がある一方、`AddFriendView` から継続使用されている。
- **影響**: 常時失敗する経路が残り、障害調査と実装判断を混乱させる。
- **根拠ファイル**:
  - `heyho_ios/Services/FirestoreService.swift`
  - `heyho_ios/Features/Profile/AddFriendView.swift`
  - `firestore.rules`

### MNT-002（P1） `createdAt` が更新処理でも上書きされる

- **問題**: `createOrUpdateUser` が `merge: true` で毎回 `createdAt: serverTimestamp()` をセットする。
- **影響**: 本来「作成日時」であるべき値が更新タイミングで変わり、監査・表示・分析で整合性が崩れる。
- **根拠ファイル**:
  - `heyho_ios/Services/FirestoreService.swift`

### MNT-003（P2） `FirestoreService` に責務が集中している

- **問題**: User / Invite / Friends / HeyHo / Query ロジックが単一クラスに混在。
- **影響**: 変更影響範囲が広く、テスト分割や段階的改修がしづらい。
- **根拠ファイル**:
  - `heyho_ios/Services/FirestoreService.swift`

### MNT-004（P2） 友だち数に比例して読み取りが増える設計

- **問題**: 行状態判定で友だちごとに最新メッセージ取得クエリを実行。
- **影響**: 友だち増加時に読み取り回数・待ち時間・コストが線形増加する。
- **根拠ファイル**:
  - `heyho_ios/Services/FirestoreService.swift`（`getFriendRowStates`, `getLastHeyHo`）

## 補足（今回の確認で改善済みと判断した点）

- `heyhos` 作成で `exists(.../friends/...)` を要求しており、送信先制約は以前より強化されている。
- `users` / `inviteCodes` で `list` を明示拒否しており、コレクション列挙は抑制されている。
- `randomNonce` の文字セット欠落は解消されている。

## 次に着手するなら（優先順）

1. `users` の公開情報と秘匿情報を分離（`fcmToken`/`inviteCode` の直接取得を不可にする）
2. 友だち関係の書き込み権限を見直し（同意モデルに合わせる）
3. 検索導線の停止 or サーバー経由検索へ置換
