# コードレビュー指摘（セキュリティ・メンテナンス）

> **対象読者**: Claude Code / 開発者  
> **目的**: 既知のリスクと保守上の注意点を一覧化し、改修・Issue 起票時の共通認識にする。  
> **注意**: 本書は「指摘の記録」であり、記載内容の実装完了を意味しない。

---

## エージェント向け：このドキュメントの使い方

1. **Firestore ルール変更・ユーザー周りの機能追加**の前に、セキュリティ章（SEC-*）を読む。
2. **新規クエリ・インデックス追加**の前に、メンテナンス章（MNT-*）のインデックス・`FirestoreService` 項を読む。
3. **修正タスクに着手する場合**: 各項目の「推奨対応の方向性」を起点に、別途 GitHub Issue 化またはユーザー確認（プロジェクトルール）を行う。

**優先度の目安**

| ラベル | 意味 |
|--------|------|
| **P0** | 本番で悪用・漏えいに直結しうる |
| **P1** | 信頼性・コスト・スパム等に効く |
| **P2** | スケール・可読性・ドキュメント整合 |

## セキュリティ（Firestore / バックエンド）

### SEC-001 — `users` コレクションの `read` が広すぎる（P0）

| 項目 | 内容 |
|------|------|
| **概要** | ログイン済みユーザーなら**任意の** `users/{userId}` ドキュメントを読める。 |
| **根拠** | `firestore.rules` の `match /users/{userId}` で `allow read: if isSignedIn();` |
| **影響** | `AppUser` が含む `fcmToken`・`inviteCode` 等が、悪意あるクライアントから取得されうる。プッシュ宛先の漏えいはハラスメント・なりすまし通知に繋がりうる。 |
| **関連コード** | `heyho_ios/Models/User.swift`（`AppUser`）、`heyho_ios/Services/FirestoreService.swift`（`getUser` / `getUsers` / `searchUsers`） |
| **推奨対応の方向性** | 公開用フィールドのみを別ドキュメントや Cloud Functions 経由で返す、または `read` を `isOwner(userId)`＋必要最小限の条件に絞る。検索は Callable 等でサーバ側集約を検討。 |

### SEC-002 — `inviteCodes` の `read` が広すぎる（P0）

| 項目 | 内容 |
|------|------|
| **概要** | サインイン済みなら `inviteCodes/{code}` を広く読める。 |
| **根拠** | `firestore.rules` の `match /inviteCodes/{code}` で `allow read: if isSignedIn();` |
| **影響** | コレクション列挙や総当たりと組み合わせて `code → userId` を突き合わせやすい。招待コードが 6 桁数字の場合探索空間が限られる。 |
| **関連コード** | `heyho_ios/Services/FirestoreService.swift`（`getUserIdByInviteCode` / `ensureInviteCode`） |
| **推奨対応の方向性** | `get` のみ許可し `list` を拒否する等、ルールで列挙を防ぐ。コード検証は Callable Function に寄せる。 |

### SEC-003 — `heyhos` 作成時に友だち関係を検証していない（P1）

| 項目 | 内容 |
|------|------|
| **概要** | 作成ルールは `fromUserId == request.auth.uid` のみ。受信者が友だちかは検証されない。 |
| **根拠** | `firestore.rules` の `match /heyhos/{heyhoId}` の `allow create` |
| **影響** | UID が分かれば**友だち以外**へもメッセージ作成可能。`functions/src/index.ts` の `onHeyHoCreated` 経由で**通知が飛ぶ**ため、スパム・嫌がらせの経路になる。 |
| **関連コード** | `firestore.rules`、`functions/src/index.ts`、`heyho_ios/Services/FirestoreService.swift`（`sendHeyHo`） |
| **推奨対応の方向性** | セキュリティルールで `exists(/databases/.../users/{uid}/friends/{friendId})` 等を要求する、または作成を Cloud Function のみにしクライアントは `call` する。 |

### SEC-004 — 友だちサブコレクションの対称書き込み（P1 ※プロダクト依存）

| 項目 | 内容 |
|------|------|
| **概要** | `users/{userId}/friends/{friendId}` に対し、`isOwner(userId) \|\| isOwner(friendId)` で read/write 可能。 |
| **根拠** | `firestore.rules` の `friends` サブコレクション |
| **影響** | **相手の同意なし**に、双方の `friends` ドキュメントを作れる設計（一方がもう一方を「友だち」にできる）。仕様として意図的ならよいが、申請・承認フローではない。 |
| **関連コード** | `heyho_ios/Services/FirestoreService.swift`（`addFriend`） |
| **推奨対応の方向性** | 承認制にするなら、片側だけ書けるルールや `friendRequests` コレクションの導入を検討。 |

### SEC-005 — 表示名更新時の `fcmToken: nil` マージ（P1・要実機/SDK 確認）

| 項目 | 内容 |
|------|------|
| **概要** | `AuthState.updateDisplayName` から `FirestoreService.createOrUpdateUser` が `fcmToken: nil` で呼ばれる。 |
| **根拠** | `heyho_ios/Auth/AuthState.swift`、`heyho_ios/Services/FirestoreService.swift` の `createOrUpdateUser`（`[String: Any]` に `fcmToken` を含める） |
| **影響** | SDK 次第で **FCM トークンが null 上書き・削除**され、プッシュが止まる可能性。セキュリティというより信頼性・サポートコスト。 |
| **推奨対応の方向性** | 表示名のみ更新する API（`displayName` だけ `merge`）に分ける、または `fcmToken` キーを省略する実装に統一。 |

### SEC-006 — Cloud Function の `messageType` 未検証（P2）

| 項目 | 内容 |
|------|------|
| **概要** | `onHeyHoCreated` が `snap.data()` の `messageType` をそのまま利用。 |
| **根拠** | `functions/src/index.ts` |
| **影響** | 想定外の文字列で通知タイトル等が意図しない分岐になる（悪用は通知文言程度）。 |
| **推奨対応の方向性** | `hey` / `ho` / `letsGo` のホワイトリストで正規化し、それ以外はデフォルトまたは無視。 |

### SEC-007 — DEBUG の匿名サインイン（P2・ビルド設定依存）

| 項目 | 内容 |
|------|------|
| **概要** | `#if DEBUG` で匿名ログインが有効。 |
| **根拠** | `heyho_ios/Auth/AuthState.swift`、`heyho_ios/Auth/SignInView.swift` |
| **影響** | 誤って本番相当ビルドに混入、または Firebase 側で匿名が有効なままだとテストアカウントの乱立等。 |
| **推奨対応の方向性** | 本番 Firebase では匿名プロバイダ無効、CI での設定確認。 |

## メンテナンス性（構造・ドキュメント・運用）

### MNT-001 — アーキテクチャドキュメントと実装のズレ（P1）

| 項目 | 内容 |
|------|------|
| **概要** | リポジトリ内の説明が古く、`yos` コレクション等の記述が残っている可能性がある。実装は `heyhos` / `inviteCodes`。 |
| **根拠** | `CLAUDE.md` 等（プロジェクト概要・データモデル節） |
| **推奨対応の方向性** | `CLAUDE.md` と `firestore.rules` を単一の真実源として同期。コレクション名・フィールドを表形式で一覧化。 |

### MNT-002 — `FirestoreService` の責務集中（P2）

| 項目 | 内容 |
|------|------|
| **概要** | ユーザー・招待・友だち・HeyHo・検索・リスナーが同一クラスに集約。 |
| **根拠** | `heyho_ios/Services/FirestoreService.swift` |
| **影響** | 変更の影響範囲が読みにくい。単体テストの分割もしづらい。 |
| **推奨対応の方向性** | ドメイン別型（例: `UserFirestoreRepository`）への分割、または `protocol` による境界の明示。 |

### MNT-003 — 友だち人数に比例するクエリ数（P2）

| 項目 | 内容 |
|------|------|
| **概要** | `getFriendRowStates` が友だちごとに `getLastHeyHo` を実行。 |
| **根拠** | `heyho_ios/Services/FirestoreService.swift` |
| **影響** | 人数増で読み取り回数・レイテンシ・課金が線形増加。 |
| **推奨対応の方向性** | 集約ドキュメント、バッチ取得設計、またはサーバ側集約の検討（仕様トレードオフあり）。 |

### MNT-004 — 複合インデックスとクエリの対応確認（P1）

| 項目 | 内容 |
|------|------|
| **概要** | `firestore.indexes.json` に `fromUserId` + `toUserId` + `createdAt` がある一方、受信クエリは `toUserId` + `createdAt`。 |
| **根拠** | `firestore.indexes.json`、`FirestoreService.inboxListener` |
| **影響** | 本番でインデックス不足エラー、またはコンソール手動作成とリポジトリの乖離。 |
| **推奨対応の方向性** | 全クエリを洗い出し、`firestore.indexes.json` と Firebase コンソールを一致させる。 |

### MNT-005 — リスナー内のエラーと空配列（P2）

| 項目 | 内容 |
|------|------|
| **概要** | `inboxListener` で `snapshot` が取れない場合、`error != nil` のときのみ `onUpdate([])`。 |
| **根拠** | `heyho_ios/Services/FirestoreService.swift` |
| **影響** | 呼び出し側でエラーと「本当に0件」を区別しづらく、UI のちらつきの原因になりうる。 |
| **推奨対応の方向性** | `Result` 型や `(items, error?)` のコールバックに変更。 |

### MNT-006 — `randomNonce` の文字セット typo（P2）

| 項目 | 内容 |
|------|------|
| **概要** | `AppleSignInHelper.swift` の nonce 用 charset に `Q` が含まれていない可能性（`...RSTUVXYZ...`）。 |
| **根拠** | `heyho_ios/Auth/AppleSignInHelper.swift` |
| **影響** | エントロピーがわずかに低下（実害は限定的）。 |
| **推奨対応の方向性** | 文字セットを修正し、コメントで意図を明示。 |

### MNT-007 — エラー表現の独自 `NSError`（P2）

| 項目 | 内容 |
|------|------|
| **概要** | `FirestoreService` がドメイン `"FirestoreService"` と数値コードでエラーを区別。 |
| **根拠** | `heyho_ios/Services/FirestoreService.swift` |
| **影響** | 横断的な `catch` やロギングの統一が難しい。 |
| **推奨対応の方向性** | `enum AppError: Error` 等に寄せる、または `LocalizedError` 準拠でメッセージを一元化。 |
